# v0.1.0 Acceptance Criteria — blueteamthreathunting

> Scope gate for all downstream specialists. Every checkbox below must be ticked before
> v0.1.0 is tagged and pushed. Resolved defaults (from PLAN.md) are locked — do not
> reopen them. Thresholds must be documented as tunable in every artifact that uses them.

## watchlists/

- [ ] `VIPUsers.csv` exists with exactly these columns (in order):
      `UserPrincipalName`, `ObjectId`, `DisplayName`, `Department`, `VIPReason`, `Notes`
- [ ] `PrivilegedAccounts.csv` exists with exactly these columns (in order):
      `UserPrincipalName`, `ObjectId`, `DisplayName`, `AssignedRole`, `RoleSource`, `Notes`
      (`RoleSource` values: `Static`, `PIM-eligible`, `PIM-active`)
- [ ] Both CSVs have at least two representative sample rows (placeholder data, clearly
      marked as examples).
- [ ] Primary join key is `UserPrincipalName`; secondary is `ObjectId` — both columns
      present and named exactly as above.
- [ ] `VIPUsers.md` companion doc exists: lists every column, states alias `VIPUsers`,
      states SearchKey `UserPrincipalName`, and shows the `_GetWatchlist("VIPUsers")` call
      pattern, plus a note that dynamic hydration from `IdentityInfo`/PIM is a future
      enhancement.
- [ ] `PrivilegedAccounts.md` companion doc exists: same requirements as above, alias
      `PrivilegedAccounts`, SearchKey `UserPrincipalName`.

## hunting-queries/

Seven YAML files, one per detection. Each file must pass the schema check below. "YAML
schema" means: `id` (UUID), `name`, `description`, `severity`, `requiredDataConnectors`,
`tactics`, `relevantTechniques`, `query`, plus optional `tuneThresholds` and
`requiresUEBA` boolean fields.

- [ ] **impossible-travel.yaml** — id present (UUID), severity High, technique T1078,
      tactic InitialAccess, query uses `SigninLogs`, speed threshold ~800 km/h documented
      in `tuneThresholds`, watchlist boost joins `_GetWatchlist("VIPUsers")` and
      `_GetWatchlist("PrivilegedAccounts")` — boosted accounts surfaced higher in output,
      `requiresUEBA: false`.
- [ ] **mfa-fatigue.yaml** — id present, severity High, technique T1110, tactic
      CredentialAccess, query uses `SigninLogs`, threshold 5 MFA denials / 10-minute
      window documented in `tuneThresholds`, watchlist boost applied, `requiresUEBA: false`.
- [ ] **dormant-admin-reactivation.yaml** — id present, severity Medium, technique T1078,
      tactic Persistence, query uses `SigninLogs` + `AuditLogs`, 30-day inactivity
      threshold documented in `tuneThresholds`, watchlist boost applied,
      `requiresUEBA: false`.
- [ ] **new-elevated-admin-role.yaml** — id present, severity High, technique T1098,
      tactic PrivilegeEscalation, query uses `AuditLogs`, watchlist boost applied,
      `requiresUEBA: false`.
- [ ] **off-hours-privileged-role-activation.yaml** — id present, severity Medium,
      technique T1078, tactic DefenseEvasion, query uses `AuditLogs` or `SigninLogs`,
      off-hours definition (outside 07:00–19:00 local + weekends) documented in
      `tuneThresholds`, watchlist boost applied, `requiresUEBA: false`.
- [ ] **vip-risky-sign-ins.yaml** — id present, severity High, technique T1078, tactic
      InitialAccess, query uses `SigninLogs` with `RiskLevelDuringSignIn`, watchlist
      boost applied, `requiresUEBA: true` (depends on Entra Identity Protection /
      `BehaviorAnalytics`).
- [ ] **mfa-auth-method-tampering.yaml** — id present, severity High, technique T1556,
      tactic CredentialAccess, query uses `AuditLogs`, watchlist boost applied,
      `requiresUEBA: false`.
- [ ] Every YAML file: `requiresUEBA: true` files include a comment in the query
      explaining graceful degradation (what to remove if UEBA tables are absent).
- [ ] Every YAML file: scoring note — accounts matching both `VIPUsers` and
      `PrivilegedAccounts` watchlists receive the highest priority score (additive boost).

## zero-trust/

Correlation queries grouped by Zero Trust pillar. Each query is a standalone `.kql` file
with a header comment that names the pillar, the Log Analytics tables used, the MITRE
technique(s), and a tuning note where thresholds apply.

- [ ] `verify-explicitly/` subfolder (or equivalent grouping) contains at least 2
      queries: one using `SigninLogs` (Conditional Access gaps / risky sign-ins) and one
      using `AuditLogs` (identity verification events).
- [ ] `least-privilege/` subfolder contains at least 2 queries: one covering
      `AuditLogs` (role assignment changes) and one covering dormant admin detection
      (cross-joining `SigninLogs` + `AuditLogs`).
- [ ] `assume-breach/` subfolder contains at least 2 queries: one covering impossible
      travel (`SigninLogs`) and one covering MFA fatigue / auth-method changes
      (`SigninLogs` + `AuditLogs`).
- [ ] Every query header names: pillar, tables, MITRE technique(s), tuning note.

## analytics-rules/

Highest-fidelity detections promoted to deployable ARM JSON. Shape must mirror
`Sentinelmonitoring/analytics-rules/entra-privileged-role-added.json` exactly
(`$schema`, `contentVersion`, `parameters.workspace`, resource type
`Microsoft.OperationalInsights/workspaces/providers/alertRules`, `kind: Scheduled`,
`tactics`, `techniques`, `entityMappings`).

- [ ] **impossible-travel.json** — severity High, queryFrequency PT1H, entityMappings
      include Account (UserPrincipalName) and IP.
- [ ] **mfa-fatigue.json** — severity High, queryFrequency PT15M, entityMappings include
      Account.
- [ ] **new-elevated-admin-role.json** — severity High, queryFrequency PT1H, entityMappings
      include Account (Actor) and Account (Target).
- [ ] **mfa-auth-method-tampering.json** — severity High, queryFrequency PT1H,
      entityMappings include Account.
- [ ] Every ARM file: GUID in resource `name` field is a valid UUID (not a placeholder
      like `00000000-0000-0000-0000-000000000000`).
- [ ] Every ARM file: `enabled: true`, `suppressionEnabled: false`, thresholds commented
      as tunable inside the `query` string.

## workbooks/

- [ ] `zt-identity-correlation.workbook.json` exists, schema key
      `"$schema": "https://github.com/Microsoft/Application-Insights-Workbooks/blob/master/schema/workbook.json"`,
      `"version": "Notebook/1.0"`.
- [ ] Workbook contains a time-range parameter (type 4, `TimeRange`).
- [ ] Workbook contains at least one tile per Zero Trust pillar (Verify Explicitly /
      Least Privilege / Assume Breach) — minimum 3 KQL grid tiles (type 3).
- [ ] Workbook includes a VIP + privileged account filter parameter linked to the
      watchlist aliases.
- [ ] Workbook tiles reference `SigninLogs` and `AuditLogs`; any tile needing
      `BehaviorAnalytics` / `IdentityInfo` includes a conditional-show note or
      inline comment marking it as UEBA-dependent.
- [ ] File parses as valid JSON (CI check covers this).
